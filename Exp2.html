Experiment 2(a): Managing a Legacy Website with RCS

Aim (5 Marks)

To initialize and manage a version control system for a legacy website project using RCS (Revision Control System), including adding files, checking in/out revisions, and viewing revision history.

Theory/Algorithm (5 Marks)

RCS (Revision Control System) is a version control tool used to manage changes to individual files. It stores versions of text files and allows developers to track changes over time, revert to previous revisions, and manage concurrent edits.

Key Steps in RCS:

  1. Initialize the RCS repository by creating an RCS/ directory.
  2. Check in files using ci (check-in command), which creates a versioned .v file.
  3. Use co (check-out command) to retrieve working copies.
  4. View version history with rlog.

Program/Command (10 Marks)

bash # Step 1: Navigate to the project directory cd legacy-website # Step 2: Create the RCS directory (if not already present) mkdir RCS # Step 3: Add a file to RCS (first-time check-in) ci -u index.html # Step 4: Make changes to the file echo "" >> index.html # Step 5: Check-in the updated file with a message ci -m"Added footer section" index.html # Step 6: Check out the file for editing again co -l index.html # Step 7: View revision history rlog index.html

Output (10 Marks)

yaml RCS/index.html,v <-- index.html enter description, terminated with single '.' or end of file: NOTE: This is the initial version. . revision 1.1 date: 2025-04-21; author: tushar; state: Exp; Initial revision revision 1.2 date: 2025-04-21; author: tushar; state: Exp; Added footer section

Result (5 Marks)

The RCS repository was successfully initialized. Files were added, revisions were managed using check-in/check-out, and revision history was viewed successfully using RCS commands.
Experiment 2(b): Bare vs Non-Bare Git Repositories

Aim (5 Marks)

To understand and explain the difference between a bare and a non-bare Git repository and their use cases in version control.

Theory/Algorithm (5 Marks)

In Git, repositories can be of two types:

Feature Bare Repository Non-Bare Repository
Contains Working Directory ❌ No working directory ✅ Has a working directory
Intended Use For collaboration/server-based sharing (e.g., origin) For local development
Git Content Structure Contains only .git/ files Contains .git/ and working files
Push/Pull ✅ Used to push/pull from remotes ❌ Should not push into directly

Bare Repository

A bare repository is essentially just the .git directory content without a working directory. It's designed to be a central repository that multiple developers can push to and pull from.

Think of it as the "server-side" repository that doesn't need working files.

Non-Bare Repository

A non-bare repository includes both the .git directory and a working directory with the actual project files. This is what developers use on their local machines.

This allows developers to edit files, stage changes, and commit locally.

Program/Command (10 Marks)

bash # Create a bare repository (for remote use) git init --bare project.git # Output: Initialized empty Git repository in /path/to/project.git/ # Create a non-bare repository (for local development) mkdir dev-project cd dev-project git init # Output: Initialized empty Git repository in /path/to/dev-project/.git/

Output (10 Marks)

bash # Bare Repo Initialized empty Git repository in /home/user/project.git/ # Non-Bare Repo Initialized empty Git repository in /home/user/dev-project/.git/

Result (5 Marks)

The differences between bare and non-bare Git repositories were successfully explained. Commands were demonstrated to initialize both types of repositories, and their respective use cases were clarified.
