Experiment 1(a): Cloning and Modifying a Git Repository

Aim (5 Marks)

To clone an existing Git repository from a remote server, make modifications to a file, commit the changes, and push them back to the remote repository.

Theory/Algorithm (5 Marks)

Git is a distributed version control system that allows developers to collaborate on projects efficiently. The process of cloning a repository involves copying an existing remote repository to a local machine. After making changes to files in the local copy, Git allows committing (saving) those changes and pushing them back to the original remote repository to update the central codebase.

Steps:

  1. Clone the remote repository using git clone <url>.
  2. Make changes to any file using a code/text editor.
  3. Add the changed file to the staging area using git add <filename>.
  4. Commit the changes using git commit -m "message".
  5. Push the commit to the remote repository using git push.

Program/Command (10 Marks)

bash # Step 1: Clone the repository git clone https://github.com/example-user/demo-repo.git # Step 2: Navigate into the repository cd demo-repo # Step 3: Edit any file (e.g., README.md) echo "This is a new line added to the README file." >> README.md # Step 4: Stage the changes git add README.md # Step 5: Commit the changes git commit -m "Updated README with a new line" # Step 6: Push the changes to the remote repository git push origin main

Output (10 Marks)

bash Cloning into 'demo-repo'... remote: Enumerating objects: 10, done. remote: Counting objects: 100% (10/10), done. remote: Compressing objects: 100% (6/6), done. Receiving objects: 100% (10/10), done. [main 6a1f3b2] Updated README with a new line 1 file changed, 1 insertion(+) To https://github.com/example-user/demo-repo.git f1e2d3a..6a1f3b2 main -> main

Result (5 Marks)

The experiment was successfully performed. The existing repository was cloned, a file was modified, changes were committed locally, and updates were pushed back to the remote repository.
Experiment 1(b): Git Tags vs Branches

Aim (5 Marks)

To understand the purpose of Git tags and how they are different from branches in a version control environment.

Theory/Algorithm (5 Marks)

Git tags are used to mark specific points in the repository's history as important, typically used for version releases (e.g., v1.0, v2.5). Unlike branches, tags do not change or move with new commits. Tags are read-only pointers to specific commits.

Differences between Tags and Branches:

Feature Tags Branches
Purpose Marking release points Feature development, bug fixes
Movability Fixed (points to a specific commit) Moves with new commits
Editability Not editable Can be updated with new commits
Usage Used for versioning Used for ongoing development

Tagging Syntax Example:

bash # Create a tag git tag v1.0 # Push a tag to remote git push origin v1.0 # List all tags git tag

Program/Command (10 Marks)

bash # Create a new tag git tag v1.0 # Verify the tag was created git tag # Push the tag to remote git push origin v1.0

Output (10 Marks)

php # Output of git tag v1.0 # Output of git push origin v1.0 To https://github.com/example-user/demo-repo.git * [new tag] v1.0 -> v1.0

Result (5 Marks)

The concept of Git tags was successfully explained and demonstrated. Git tags are used to label specific versions, unlike branches which continue to evolve over time.
